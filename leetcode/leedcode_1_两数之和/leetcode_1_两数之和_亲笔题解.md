```
using namespace std;
class Solution{
	public:
		vector<int> twoSum(vector<int> &num,int target){
	unordered_map<int,int> mp;
	for(int i=0;i<num.size();i++){
		int x=num[i];
		int need=target-x;
		if(mp.count(need)){
			return {mp[need],i};
		}
        if(!mp.count(x)){
            mp[x]=i;
        }
		
	}
    return {};
}
};


```

## 洛谷转力扣的阵痛

洛谷会给清楚输入输出,要求写的是一个完整的程序

力扣则是平台要求OOP,已经写死约定好了函数名,param个数和类型,所以意思就是说,力扣大多数情况下都是只需要写函数方法,而非整个完整程序,这就是我被卡好多次的原因



## unordered_map的坑

第一次用,一定要注意,mp.count(x)或者mp.find(x)才能判断x是否存在,

如果是mp[x],如果x不存在,会自动新建它的值为0,这是我们都不想看到的

所以要注意,count,和find才能判断元素是否存在



## 哈希表初识

复杂度O(n),写这题就是为了学习哈希表,unordered_map的,如果还是停在两重循环的境界,真的是太杜烂了

核心就是空间换时间

unordered_map核心就是一个无序的map,复杂度在OI中可以视作O(1),虽然最坏情况是O(n),存在一个哈希命中的概率问题,不过这不是要理解的重点;

unordered_map本质就是键值对,本题里键就是数组里的数字,值就是这个数字在数组里的索引角标,因为我们要求的就是角标,所以角标就是值;

明确了键值对关系之后,确定一个x,然后need就可以知道了,那么一开始mp是个空白的unorderedmap,在步进的过程中,如果mp.count(need)不为0那就万事大吉,直接return,

如果是0,意味着还没标记到这里或者数组里根本不存在need这个数,那只能继续挑选下一个x,那么由于mp百废俱兴,所以我们要把一切能利用的利用上,既然已经知道第i个元素是x了,那么就可以直接标注mp[x]=i,当然,前提是x还没标记过,如果标记过就不用标记了,本质上题目前提也规定了不会有这种情况出现;

这样子的效果就是当下为未来可能的查找布局,就不会导致未来想找的时候mp.count(need)永远是0,

## 边界的警觉

如果[3,3],target=6,这会不会用到同一个元素呢?不会,核心在于我们的思想是**<u>先查再存</u>**, 因此当前查询的元素都是之前的,而不是当下的

步进演示:i=0,x=3,need=3,mp.count(need)=0,->mp[3]=0

i=1,x=3,need=3,发现mp[3]=0,因此返回(mp[3],1),即(0,1)

**<u>考虑边界情况有时候会决定算法的正误,比赛的胜负,用笔演算有时候是好方法</u>**
